# 跨平台客户端共用代码方案

## 目标

希望同一套客户端能力覆盖以下平台：

- iOS
- Android
- Windows
- Linux

核心诉求是：

1. **尽量共用同一套业务代码**
2. **不同平台使用不同的打包与发布方式**
3. 保持当前设备端播放器能力可持续演进
4. 为后续平台扩展预留结构，而不是每个平台各写一套

---

## 结论

**可以共用大部分代码，但不建议追求 100% 完全共用。**

更合理的方式是：

- **共用一套前端核心代码**：页面、组件、状态管理、接口调用、播放逻辑
- **按平台分别提供壳层**：桌面端壳、移动端壳
- **按平台分别打包发布**：Windows / Linux、iOS / Android 使用各自的构建链

也就是：

> **共享核心 + 平台适配层 + 各平台独立打包**

---

## 结合当前项目的推荐路线

当前项目已经有 `device-client`，并且是：

- Vue
- Pinia
- Vue Router
- Electron

这意味着最现实的演进方向是：

### 推荐技术路线

- **共享 UI / 业务层**：Vue 3
- **桌面端**：Electron（优先，迁移成本最低）
- **移动端**：Capacitor（封装同一套 Vue 页面）

如果后续希望桌面包更轻，也可以再评估 Tauri，但第一阶段不建议一边跨平台一边迁移桌面框架。

---

## 哪些代码可以共用

以下内容通常可以高度共用：

### 1. 页面与组件
- 播放器页面
- 列表页
- 配置页
- 登录/绑定页
- 通用 UI 组件

### 2. 状态管理
- 当前设备状态
- 当前播放媒体状态
- 分发列表状态
- 同步状态
- 错误状态

### 3. 业务逻辑
- 媒体轮播逻辑
- 图片/视频切换逻辑
- 自动下一项逻辑
- BGM 控制逻辑
- 同步拉取逻辑
- 鉴权地址解析逻辑

### 4. 网络与数据层
- API 封装
- 请求拦截
- 数据模型
- DTO 转换
- 错误码处理

### 5. 大部分样式
- 播放区域布局
- 侧栏布局
- 通用主题与视觉风格

---

## 哪些内容需要平台适配

这些内容通常不能完全共用，需要单独封装：

### 1. 设备能力
- 设备唯一标识
- 主机名/设备名
- 系统信息获取
- 屏幕信息

### 2. 系统集成能力
- 开机自启动
- 全屏 / 无边框 / Kiosk 模式
- 系统托盘
- 系统通知
- 窗口管理

### 3. 文件与媒体能力
- 本地文件访问
- 缓存目录
- 媒体文件下载与持久化
- 权限申请

### 4. 生命周期能力
- App 前后台切换
- 锁屏/休眠行为
- 自动恢复播放
- 后台同步策略

### 5. 平台限制差异
- iOS 自动播放限制
- iOS 后台执行限制
- Android 后台与前台服务机制
- Windows/Linux 对常驻展示类应用更友好

---

## 平台适配判断

### Windows
非常适合当前这类设备端播放器：
- 全屏展示友好
- 可做 kiosk
- Electron 支持成熟
- 容易接入设备能力

### Linux
同样适合：
- 适合盒子、工控机、大屏终端
- 可控性强
- 很适合作为无人值守播放端

### Android
可行性较高：
- 更适合做展示终端、广告屏、平板端播放器
- 对持续播放和设备部署比 iOS 更友好
- 可通过 Capacitor + 原生插件方式补齐能力

### iOS
可做，但限制最多：
- 后台能力受限
- 自动播放策略严格
- 不适合完全照搬桌面端终端播放器能力
- 更适合作为“轻量查看/控制端”，不一定适合作为无人值守大屏终端

## 推荐架构

建议逐步演进成 Monorepo 结构，拆出共享核心。

### 推荐目录结构

```text
cloud-album-publisher/
├─ apps/
│  ├─ desktop/                 # Windows / Linux 桌面壳（Electron）
│  └─ mobile/                  # iOS / Android 壳（Capacitor）
├─ packages/
│  ├─ app-core/                # 共享前端应用核心
│  │  ├─ src/
│  │  │  ├─ components/
│  │  │  ├─ views/
│  │  │  ├─ stores/
│  │  │  ├─ router/
│  │  │  ├─ api/
│  │  │  ├─ composables/
│  │  │  └─ utils/
│  ├─ platform-bridge/         # 平台能力抽象层
│  └─ shared-types/            # 共用类型定义
├─ backend/
├─ frontend/
└─ device-client/
```

## 推荐分层

### 第一层：共享核心层
职责：
- 页面
- 组件
- Store
- API
- 路由
- 播放逻辑
- 样式主题

目标：
- 尽量不感知运行平台

### 第二层：平台桥接层
职责：
- 提供统一接口给核心层调用
- 屏蔽 Electron / Capacitor / Web 差异

例如定义统一接口：

```ts
interface DeviceBridge {
  getDeviceIdentity(): Promise<DeviceIdentity>
  getAppVersion(): Promise<string>
  enterFullscreen(): Promise<void>
  exitFullscreen(): Promise<void>
  getCacheDir(): Promise<string | null>
}
```

然后分别实现：
- `deviceBridge.electron.ts`
- `deviceBridge.capacitor.ts`
- `deviceBridge.web.ts`

### 第三层：平台壳层
职责：
- 初始化运行环境
- 注入平台 bridge
- 打包发布
- 处理原生权限和生命周期

## 为什么不建议继续把所有代码都堆在当前 `device-client`

因为如果以后直接在当前 Electron 项目里硬加 Android/iOS 适配，会出现问题：

1. Electron 代码和通用业务代码耦合越来越深
2. 移动端需要能力时不好抽离
3. 平台判断散落在页面里
4. 后面越改越难维护

所以更好的方向是：

- **把可复用的代码先抽出来**
- **把 Electron 相关代码留在桌面壳中**
- **移动端只接自己的壳和插件**

---

## 对当前项目的迁移建议

### 阶段 1：先不改功能，只整理结构
目标：不影响现有 Windows/Linux 客户端继续运行。

建议动作：
1. 抽离 `device-client/src` 中可复用的页面、组件、store、api
2. 新建共享包，例如 `packages/app-core`
3. `apps/desktop` 继续使用 Electron，引用共享核心
4. 保证桌面端行为和当前一致

### 阶段 2：补平台桥接层
目标：把 Electron 特有调用从页面中移出去。

建议动作：
1. 定义统一 `DeviceBridge` 接口
2. 把设备标识、窗口行为、缓存目录等能力改为通过 bridge 获取
3. 在桌面端实现 Electron bridge

### 阶段 3：新增移动端壳
目标：让 Android/iOS 可以加载共享核心。

建议动作：
1. 新建 `apps/mobile`
2. 使用 Capacitor 集成 Vue 构建产物
3. 实现 Capacitor bridge
4. 先跑通登录、拉取、列表、图片播放
5. 再逐步补齐视频、缓存、后台等高级能力

### 阶段 4：按平台差异做能力收敛
目标：避免强行让所有平台功能完全一致。

建议思路：
- Windows/Linux：完整终端播放器能力
- Android：尽量接近桌面能力
- iOS：优先做简化版或受限版

## 关于“是否真的一套代码通吃”

现实里最接近的表达应该是：

- **70% ~ 90% 业务代码共用**
- **10% ~ 30% 平台适配代码独立维护**

如果项目对设备能力依赖越强，这个“不可共用”的比例就会更高。

你的这个项目属于“设备播放器/终端客户端”类型，通常：

- UI 和业务层很适合共用
- 平台能力层一定要分开

## 对本项目的最终建议

### 推荐优先级
1. **先做好 Windows / Linux 共用桌面端**
2. **再扩展 Android**
3. **最后评估 iOS 是否做完整播放器端，还是做轻量版**

### 推荐原因
- 当前项目已经有 Electron 基础，最容易先沉淀共享层
- Android 更适合作为展示终端延展平台
- iOS 限制较多，越晚做决策越稳妥

## 简短结论

这件事**可行**，而且适合你当前项目的路线是：

- **共享 Vue 业务核心**
- **桌面端继续 Electron**
- **移动端新增 Capacitor 壳**
- **通过平台桥接层处理差异**

不要追求所有平台完全一样，而是追求：

> **大部分代码共用，平台能力分层隔离，分别打包发布。**
