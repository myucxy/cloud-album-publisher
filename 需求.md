
# 电子相册发布系统设计文档

## 1. 项目概述

### 1.1 项目名称

**智控云影——电子相册智能发布系统**

### 1.2 项目目标

构建一个支持多平台、多媒体内容管理的电子相册发布系统，实现照片/视频上传、编辑、分享及多终端内容分发管理。

### 1.3 系统范围

**包含内容：**

- 图片/视频上传与处理
- 电子相册生成
- 背景音乐添加
- 多终端客户端管理
- 内容分发控制

**不包含内容：**

- 硬件设备制造
- 操作系统开发
- 基础网络设施

## 2. 用户角色与权限

### 2.1 用户角色定义

- **普通用户：** 上传照片/视频、创建相册、添加BGM、管理个人内容
- **管理员：** 管理用户账户、审核内容、配置系统参数、监控系统状态
- **客户端设备：** 作为内容接收端，需支持绑定到用户账户

### 2.2 权限管理需求

- 基于角色的访问控制（RBAC）
- 用户可设置相册权限（公开/私有/指定设备可见）
- 管理员拥有系统级管理权限

## 3. 功能需求

### 3.1 核心内容管理功能

- **图片/视频上传：** 支持批量上传、多种格式（JPG、PNG、MP4等）、大文件传输
- **电子相册生成：** 首期支持基础排版、封面设置、播放顺序与简单播放参数；自定义模板作为后续扩展
- **BGM添加：** 首期支持本地音乐文件上传、音量调节；在线音乐库作为后续扩展
- **内容编辑：** 首期仅支持基础元数据编辑与封面裁剪；复杂滤镜、视频剪辑放入后续阶段

### 3.2 多终端管理功能

- **客户端绑定：** 将不同设备绑定到用户账户
- **内容分发控制：** 为每个绑定设备设置可访问的相册/视频内容
- **设备分组管理：** 支持创建设备组，批量设置内容权限

### 3.3 系统管理功能

- **用户管理：** 注册、登录、密码找回、个人信息维护
- **内容审核：** 管理员审核用户上传内容
- **数据统计：** 用户活跃度、内容访问量等统计分析

## 4. 非功能需求

### 4.1 性能需求

- **照片上传速度：** ≤5秒/张（1080P分辨率）
- **视频上传速度：** ≤30秒/分钟（1080P分辨率）
- **系统响应时间：** ≤1秒（关键操作）
- **支持并发用户数：** ≥1000

### 4.2 安全性需求

- **数据加密传输：** HTTPS/TLS协议
- **用户隐私保护：** 符合GDPR等数据保护法规
- **内容安全：** 敏感内容过滤机制
- **定期数据备份：** 防止数据丢失

### 4.3 可扩展性需求

- **模块化设计：** 便于后续功能扩展
- **支持存储扩容：** 适应用户增长
- **API接口开放：** 支持第三方系统集成

### 4.4 多终端适配需求

- **跨平台支持：** TV、安卓、iOS、Windows、Mac、Linux、ARM等
- **响应式设计：** 自动适应不同屏幕尺寸
- **设备特性优化：** 针对不同设备优化交互方式

## 5. 技术架构设计

### 5.1 后端技术选型

**首期落地方案：Java + Spring Boot 服务端优先、模块化单体架构**

- **优先级原则：**
    - 先完成服务端核心能力，再逐步接入Web、移动端、TV端
    - 服务端作为唯一业务真源，负责鉴权、媒体元数据、分发规则、任务编排
    - 首期避免过早拆分微服务，先以模块化单体降低协作和部署成本
- **语言选择理由：**
    - 企业级稳定性
    - 并发处理能力
    - 技术人才储备丰富
    - 适合后续从单体平滑演进到微服务
- **核心框架：**
    - **Spring Boot 3.x：** 核心框架，提供自动配置、嵌入式服务器
    - **Spring Security：** 统一认证鉴权
    - **MyBatis-Plus：** ORM框架，适合复杂查询场景
    - **Redis：** 用于缓存热点数据、上传进度记录、限流
    - **OpenAPI/Swagger：** 统一接口契约，支撑多人并行开发

### 5.2 系统架构分层

**1. 接入层：**

- **统一HTTP API：** 对Web管理端、用户端、设备端暴露统一REST API
- **鉴权与限流：** 在服务端入口统一处理JWT鉴权、请求追踪、限流
- **SSL终止：** 在反向代理或入口网关处理HTTPS

**2. 应用层（首期同一服务内的业务模块）：**

- **用户与认证模块：** 注册、登录、Token刷新、RBAC
- **相册模块：** 相册创建、元数据维护、封面/BGM配置
- **媒体模块：** 上传、存储、转码、缩略图、媒体状态查询
- **设备模块：** 设备绑定、心跳、在线状态、设备分组
- **分发模块：** 相册到设备的授权与播放配置
- **审核与审计模块：** 内容审核、操作日志、异常追踪
- **任务模块：** 异步媒体处理、重试、失败告警

**3. 数据层：**

- **MySQL 8.0：** 存储结构化数据
- **Redis 7.0：** 缓存热点数据、上传进度、短期任务状态
- **对象存储：** 阿里云OSS/腾讯云COS/MinIO，存储原始图片/视频文件

**4. 基础设施层：**

- **Docker：** 容器化部署服务端与依赖
- **CI/CD：** 自动构建、测试、发布
- **Prometheus + Grafana：** 监控系统性能指标
- **集中日志：** 支撑问题排查和多人协作定位

### 5.3 架构演进路线

**阶段1（建议优先实现）：** 模块化单体服务，完成业务闭环和接口契约稳定。

**阶段2：** 将媒体处理任务独立为异步Worker服务，缓解CPU密集型任务压力。

**阶段3：** 当团队规模、并发量、发布频率明显上升后，再按领域拆分微服务并引入服务网关。

### 5.4 技术亮点与创新

- **多媒体处理流水线：** FFmpeg实现视频转码、图片压缩，RabbitMQ消息队列解耦处理流程
- **智能相册生成：** OpenCV进行人脸检测，自动添加主题标签
- **多终端适配引擎：** 根据客户端设备类型自动调整内容格式
- **断点续传机制：** Redis记录上传进度，支持大文件稳定上传

## 6. RESTful API接口设计

### 6.1 设计原则

- **资源导向设计**
- **无状态通信**
- **统一接口**
- **可缓存性**
- **分层系统**

### 6.2 API版本控制策略

- **URL版本控制：** `/api/v1/`作为基础路径
- **媒体类型版本控制：** `Accept: application/vnd.photoalbum.v1+json`
- **版本演进规则：** 主版本、次版本、修订版本区分

### 6.3 核心资源接口

#### 用户资源（/api/v1/users）

- GET /users：获取用户列表（管理员权限）
- GET /users/{id}：获取用户详情
- POST /users：创建新用户（注册）
- PUT /users/{id}：全量更新用户信息
- PATCH /users/{id}：部分更新用户信息
- DELETE /users/{id}：删除用户（需管理员权限）

#### 相册资源（/api/v1/albums）

- GET /albums：获取用户所有相册（支持分页）
- GET /albums/{id}：获取相册详情（含内容列表）
- POST /albums：创建新相册
- PUT /albums/{id}：更新相册元数据
- DELETE /albums/{id}：删除相册（级联删除内容）
- GET /albums/{id}/contents：获取相册内容列表
- POST /albums/{id}/contents：向相册添加内容
- DELETE /albums/{id}/contents/{contentId}：从相册移除指定内容
- PATCH /albums/{id}/cover：更新相册封面图

#### 媒体资源（/api/v1/media）

- GET /media：获取当前用户媒体列表（支持分页、状态筛选、类型筛选）
- POST /media/upload/init：初始化图片/视频分片上传，返回 `uploadId/objectKey/partSize/totalParts`
- PUT /media/upload/{uploadId}/{partNumber}：上传指定分片
- POST /media/upload/{uploadId}/complete：完成分片合并并创建媒体记录
- GET /media/upload-status/{uploadId}：查询上传会话与分片进度
- GET /media/{id}：获取媒体文件（原始/缩略图）
- DELETE /media/{id}：删除媒体文件
- POST /media/{id}/process：触发媒体处理
- GET /media/{id}/status：查询媒体处理状态

#### 设备资源（/api/v1/devices）

- GET /devices：获取用户绑定的所有设备
- POST /devices：绑定新设备（需设备ID和类型）
- POST /devices/token：为已绑定设备签发设备访问令牌
- DELETE /devices/{id}：解绑设备
- PATCH /devices/{id}/name：修改设备名称
- PATCH /devices/{id}/status：更新设备状态（在线/离线）
- GET /devices/pull?deviceUid={deviceUid}：用户在已登录态下按绑定关系拉取指定设备可访问的生效分发内容，并更新最近心跳时间
- GET /devices/pull/current：设备使用自身访问令牌拉取当前设备可访问的生效分发内容，并更新最近心跳时间
- GET /devices/groups：获取当前用户设备组列表
- POST /devices/groups：创建设备组
- PATCH /devices/groups/{id}：更新设备组信息
- DELETE /devices/groups/{id}：删除设备组
- POST /devices/groups/{id}/members：向设备组添加成员
- DELETE /devices/groups/{id}/members/{deviceId}：移除设备组成员

#### 内容分发资源（/api/v1/distributions）

- GET /distributions：获取所有内容分发规则（支持分页与 `status` 状态筛选）
- GET /distributions/{id}：获取特定分发规则详情
- POST /distributions：创建新分发规则（绑定相册与设备/设备组）
- PUT /distributions/{id}：更新分发规则
- DELETE /distributions/{id}：删除分发规则
- PATCH /distributions/{id}/activate：生效分发规则
- PATCH /distributions/{id}/disable：停用分发规则
- GET /distributions/active：获取当前生效的分发规则

### 6.4 请求与响应规范

**请求头规范：**

- `Authorization: Bearer {token}`：JWT认证令牌
- `Content-Type: application/json`：JSON格式请求体
- `X-Device-Type: android/ios/web/tv`：客户端设备类型标识
- `X-Device-ID: {unique_id}`：客户端唯一标识

**响应格式：**

```
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1712320806000
}
```

**错误响应：**

```
{
  "code": 400,
  "message": "参数错误：设备ID格式不正确",
  "details": ["设备ID必须为16位十六进制字符串"],
  "timestamp": 1712320806000
}
```

### 6.5 HTTP状态码使用规范

| 状态码 | 含义 | 使用场景 |
| ------ |------ |------ |
| 200 | OK | GET/PUT/PATCH请求成功 |
| 201 | Created | POST创建资源成功 |
| 204 | No Content | DELETE请求成功，无返回数据 |
| 400 | Bad Request | 参数格式错误、缺失必传参数 |
| 401 | Unauthorized | 未登录、Token失效 |
| 403 | Forbidden | 已认证但无权限 |
| 404 | Not Found | 资源不存在 |
| 409 | Conflict | 资源冲突 |
| 429 | Too Many Requests | 请求频率过高，需限流 |
| 500 | Internal Server Error | 服务器内部错误 |

## 7. 数据库设计

### 7.1 核心数据表

- **用户表：** 存储用户基本信息、账户状态
- **相册表：** 存储相册元数据（名称、创建时间、权限设置）
- **媒体表：** 存储图片/视频信息（路径、格式、大小、处理状态）
- **相册内容关联表：** 存储相册与媒体的顺序关系、展示参数
- **设备表：** 存储绑定设备信息（设备ID、类型、绑定时间）
- **设备组表：** 存储设备分组信息
- **分发规则表：** 存储相册-设备/设备组的授权与播放配置
- **媒体处理任务表：** 存储转码、压缩、缩略图生成等异步任务
- **审核记录表：** 存储内容审核结果、原因、审核人
- **操作审计表：** 存储关键管理操作日志

### 7.2 数据关系

- 一个用户可创建多个相册
- 一个相册包含多张图片/视频
- 一个用户可绑定多个设备
- 一个设备可访问多个相册（根据权限设置）

### 7.3 关键状态模型

- **媒体状态：** `UPLOADING` -> `UPLOADED` -> `PROCESSING` -> `READY` / `FAILED`
- **分发状态：** `DRAFT` -> `ACTIVE` -> `DISABLED`
- **设备状态：** `UNBOUND` / `OFFLINE` / `ONLINE`
- **审核状态：** `PENDING` / `APPROVED` / `REJECTED`

以上状态需要在接口文档、数据库字段、前端显示文案中保持一致，避免多人开发时出现语义偏差。

## 8. 安全设计

### 8.1 认证机制

- **JWT认证：** 用户登录后获取Access Token和Refresh Token
- **PKCE增强：** Proof Key for Code Exchange增强OAuth2.0安全性

### 8.2 权限控制

- **基于角色的访问控制（RBAC）**
- **基于资源的权限控制**
- **Spring Security实现细粒度权限控制**

### 8.3 数据安全

- **HTTPS强制传输加密**
- **敏感数据加密存储：** BCrypt加密
- **媒体文件保护：** Nginx反向代理+防盗链+动态Token签名URL
- **输入验证：** Hibernate Validator防止SQL注入/XSS攻击

## 9. 关键接口实现细节

### 9.1 大文件上传接口设计

**当前实现进度（2026-04-11）：部分完成**

- 后端已落地 `POST /media/upload/init`、`PUT /media/upload/{uploadId}/{partNumber}`、`POST /media/upload/{uploadId}/complete`、`GET /media/upload-status/{uploadId}`，并提供 `GET /media` 媒体列表接口。
- 前端媒体管理页已按上述真实接口完成对齐，支持分片上传、合并提交、上传后刷新媒体列表。
- 媒体异步处理链路已可对图片执行真实读取、分辨率提取与 JPEG 缩略图生成；视频/音频已接入 FFprobe 元数据提取，并通过 FFmpeg 执行最佳努力转码、视频封面抽帧与音频封面提取，命令失败时会回退为保留已解析元数据后继续沿用现有任务状态机。
- 文档主接口已同步为当前真实实现；分片完整性校验已落地，文件级 MD5 等更强校验仍待补充。

**分片上传流程：**

1. 客户端发起`POST /media/upload/init`
2. 客户端分片上传：`PUT /media/upload/{uploadId}/{partNumber}`
3. 服务端验证分片完整性（当前已校验分片齐全，MD5校验暂未落地）
4. 客户端发起`POST /media/upload/{uploadId}/complete`

**断点续传支持：**

- Redis记录上传进度：`upload:{uploadId}:progress`
- 查询上传状态：`GET /media/upload-status/{uploadId}`

### 9.2 多终端内容分发控制

**当前实现进度（2026-04-11）：部分完成**

- 后端已支持分发规则创建、更新、删除、激活、停用、按状态分页筛选，以及当前生效规则查询。
- 分发规则现已同时支持直接绑定设备 `deviceIds` 与设备组 `groupIds`；设备拉取内容时会同时解析直绑设备与所属设备组命中的 `ACTIVE` 分发规则。
- 分发激活前已校验设备归属、设备组归属、相册内容存在、媒体状态为 `READY`、最新审核记录为 `APPROVED`，避免未就绪或未审核通过内容下发。
- 前端已提供分发规则管理页与设备管理页，支持设备绑定/解绑、重命名、状态更新、设备组 CRUD、成员增删，以及在分发规则中录入设备组 ID。
- 后端已同时提供两条设备拉取链路：`GET /devices/pull` 供用户在已登录态下按 `deviceUid` 代设备拉取内容，`GET /devices/pull/current` 供设备使用 `POST /devices/token` 签发的设备访问令牌直接拉取当前设备命中的 `ACTIVE` 分发规则；两条链路都会刷新设备心跳并仅返回 `READY` 且审核通过的媒体。
- 已新增后端高价值集成测试，覆盖分发激活校验、设备直绑/设备组命中拉取、`DEVICE_ONLY` 可见性、媒体过滤与设备心跳刷新等关键业务规则，以及媒体处理任务成功/重试/失败状态流转、自动提交审核与缺失媒体失败场景。
- 仍未完成：非功能项，以及更大范围的自动化测试补齐（如真实 MinIO/FFmpeg/FFprobe 外部依赖链路与更完整的接口/端到端覆盖）。

**分发规则模型：**

```
{
  "id": "dist-123",
  "albumId": "album-456",
  "deviceIds": ["device-789", "device-012"],
  "playbackSettings": {
    "loop": true,
    "shuffle": false,
    "duration": 10
  },
  "active": true
}
```

**设备类型适配策略：**

- **TV端：** 生成1080P/4K分辨率内容
- **手机端：** 提供触控优化界面
- **桌面端：** 支持多窗口操作
- **ARM设备：** 优化CPU使用率

### 9.3 电子相册生成接口

**创建相册流程：**

1. 客户端提交`POST /albums`
2. 服务端生成相册ID和初始结构
3. 客户端通过`POST /albums/{id}/contents`添加内容
4. 服务端自动应用主题模板

**BGM集成：**

- 支持上传本地音乐文件或选择平台音乐库
- 通过`PATCH /albums/{id}/bgm`设置背景音乐
- 支持音量调节、淡入淡出效果设置

## 10. 性能优化策略

### 10.1 媒体处理优化

- **缩略图预生成：** ImageMagick或Java Graphics2D预生成多尺寸缩略图
- **视频转码：** FFmpeg实现H.264/H.265编码
- **WebP格式支持：** 自动转换为WebP格式

### 10.2 缓存策略

- **CDN缓存：** 媒体文件通过CDN分发
- **Redis缓存：** 缓存热点相册数据
- **浏览器缓存：** ETag和Last-Modified头

### 10.3 数据库优化

- **索引优化：** 为相册表、媒体表、分发表建立复合索引
- **慢查询治理：** 先通过SQL审查、执行计划、必要索引解决性能问题
- **归档策略：** 对审计日志、处理任务日志制定归档和清理策略
- **扩展原则：** 在单库瓶颈被数据证实前，不提前引入分库分表和读写分离

## 11. 部署与运维

### 11.1 部署方案

- **首期部署：** 使用 Docker Compose 部署服务端、MySQL、Redis、对象存储/其替代方案
- **环境划分：** 至少区分开发环境、共享测试环境、生产环境
- **发布策略：** 采用镜像版本化发布，保留可回滚版本
- **扩展预案：** 当共享测试和生产环境压力上升后，再评估Kubernetes与服务拆分

### 11.2 监控与日志

- **全链路日志：** 记录请求来源、参数、响应时间
- **实时监控：** 监控API调用量、任务堆积、设备在线数、上传失败率
- **告警机制：** 对接口错误率、任务失败率、存储空间不足进行告警
- **安全审计：** 定期进行渗透测试

### 12.4 多人协同开发说明

为支持多人并行推进，文档和工程管理应遵循以下约束：

| 协同主题 | 约束要求 |
| ------ | ------ |
| 模块分工 | 至少拆分为认证、相册、媒体、设备、分发、管理端六个模块，每个模块明确负责人 |
| 接口协同 | 任何接口变更先改OpenAPI，再改代码；前后端与设备端均以接口契约为准 |
| 数据库变更 | 所有DDL通过迁移脚本管理，禁止口头同步字段变更 |
| 分支策略 | 采用`main`、`develop`、`feature/*`分支，合并前必须经过代码评审 |
| 联调环境 | 至少提供一个共享测试环境和一套可复现的本地启动说明 |
| 任务边界 | 异步任务、上传协议、状态枚举统一沉淀在公共文档，禁止各端自行约定 |
| 完成标准 | 功能代码、接口文档、数据库迁移、基础测试、变更说明缺一不可 |

### 12.5 三人分工建议

为保证首期可以并行推进，建议三人按“核心主链路 + 支撑能力 + 管理与联调”拆分：

| 成员 | 负责模块 | 主要职责 | 关键交付物 | 依赖关系 |
| ------ | ------ | ------ | ------ | ------ |
| A（后端主程） | 认证、用户、相册 | 搭建Spring Boot主工程、统一异常处理、JWT/RBAC、用户接口、相册接口、OpenAPI骨架 | 可运行服务骨架、认证接口、相册接口、基础单元测试 | 先行定义公共规范，其他两人依赖其接口骨架 |
| B（媒体与设备） | 媒体、设备、异步任务 | 分片上传、媒体表设计、FFmpeg处理任务、设备绑定/心跳、设备组接口 | 上传链路、媒体状态流转、设备接口、任务处理模块 | 依赖A提供认证和公共中间件 |
| C（分发与管理） | 分发、审核、审计、联调 | 分发规则接口、审核流、操作日志、管理端最小接口、测试环境联调、接口验收 | 分发与审核接口、审计日志、联调说明、集成测试用例 | 依赖A的基础工程与B的媒体/设备数据结构 |

**协作约束：**

1. A负责维护公共工程约定：错误码、分页结构、鉴权方式、OpenAPI主文档。
2. B和C不得绕过公共约定自行定义状态码、响应结构或数据库命名。
3. 每次接口变更都要同步更新OpenAPI与示例请求响应。
4. 每人至少负责自己模块的数据库迁移、基础测试和联调说明，不把收尾工作留给别人。

### 12.6 三人并行开发顺序

1. **第1阶段：** A先提交工程骨架、认证中间件、统一返回结构、基础OpenAPI。
2. **第2阶段：** B并行完成媒体上传和设备接口，C并行完成分发规则和审核流设计。
3. **第3阶段：** 三人共同完成共享测试环境联调，验证“上传 -> 入册 -> 分发 -> 设备可见”的主链路。
4. **第4阶段：** C主导验收清单，A和B补齐缺失文档、测试、迁移脚本。

### 12.7 特别提醒

1. **避免过度RESTful：** 复杂操作可适当使用POST
2. **多终端适配挑战：** 采用"设备类型-功能支持矩阵"
3. **服务端优先原则：** 前端页面、设备固件、管理台均不得绕过服务端直连核心数据
4. **性能瓶颈预估：** 提前规划媒体处理集群扩展能力
5. **法律合规性：** 确保用户隐私保护符合法规
6. **API文档维护：** Swagger UI生成实时API文档
